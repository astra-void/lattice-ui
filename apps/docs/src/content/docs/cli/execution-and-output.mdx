## Interactive vs Non-Interactive

By default, commands can run interactively when stdin/stdout are TTY-capable.

- Interactive mode prompts for missing selection/path input.
- `--yes` disables prompts and uses explicit flags/defaults.
- If prompts are required but no TTY is available, commands fail with guidance to use `--yes`.

## `create` Input Rules

- `project-path` is optional in interactive mode.
- In `--yes` mode, `project-path` is required.
- `--lint` enables lint template application.
- `--no-lint` disables lint template application.
- `--lint` and `--no-lint` cannot be used together.

## Structured Output Phases

Most dependency commands (`add`, `remove`, `upgrade`, `doctor`) use consistent output phases:

- `Selecting`: command scope and selected target context.
- `Planning`: expanded components/packages and summary counts.
- `Dry Run`: simulated package manager commands when `--dry-run` is set.
- `Applying`: real package manager operations (plus confirmation where applicable).
- `Result`: outcome summary and success/warning/error state.
- `Next Steps`: follow-up commands (typically `lattice doctor` or `pnpm lattice doctor`).

## Dry Run Contract

In dry-run mode:

- Planned package manager actions are printed with `[dry-run]` prefix.
- No dependency changes are applied.
- Output explicitly includes: `No files were changed.`

Examples:

### Global CLI

```bash
lattice add dialog,toast --preset overlay --dry-run
lattice remove --preset overlay --dry-run
lattice upgrade --preset form --dry-run
```

### Monorepo Local Script

```bash
pnpm lattice add dialog,toast --preset overlay --dry-run
pnpm lattice remove --preset overlay --dry-run
pnpm lattice upgrade --preset form --dry-run
```
